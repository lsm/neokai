# 01 — The Harness Problem

**Resource:** Can Bölük, “I Improved 15 LLMs at Coding in One Afternoon. Only the Harness Changed.”  
**Source:** https://blog.can.ac/2026/02/12/the-harness-problem/  
**Type:** article  
**Focus:** edit-tool design, harness benchmarking, token savings from fewer retries

## Core thesis

Can Bölük argues that AI coding performance is not determined by model quality alone. The harness — tool schemas, edit primitives, state handling, failure modes, and context mediation — is a major hidden variable. Many apparent model failures are actually expression failures: the model may know the fix, but the tool interface asks it to express the fix in a brittle format.

The article reframes coding-agent evaluation from “which model is best?” to “which interface lets each model reliably act on code?” This matters because the harness supplies input tokens, mediates every file mutation, handles failed edits, and often causes expensive retry loops.

## Key findings

The article’s strongest empirical claim is that changing only the edit harness improved success across **16 models** and **3 edit tools**. Benchmark setup used **3 runs per task**, **180 tasks per run**, fresh agent sessions, and a temporary workspace. Bugs were generated by mechanical mutations in random React files: operator swaps, boolean flips, off-by-one errors, removed optional chains, and renamed identifiers.

Reported results:

- `patch` editing was worst for nearly every model.
- Proposed `hashline` editing matched or beat exact string replacement for most models.
- Weaker models improved most.
- **Grok Code Fast 1:** **6.7% → 68.3%**.
- **MiniMax:** more than doubled.
- **Grok 4 Fast:** output tokens dropped **61%**, attributed to fewer failed-edit retry loops.
- A **+8% Gemini** improvement from harness changes exceeded many model-upgrade deltas.
- Total benchmark cost was about **$300**, with **zero training compute**.

The article also cites broader evidence: Aider benchmark results where edit format moved **GPT-4 Turbo from 26% to 59%**, while **GPT-3.5** reached only **19%** with the same format. Conclusion: format matters greatly, but model capability still matters.

## Technique: hashline editing

Hashline adds stable, verifiable line anchors to file reads and greps. Example:

```text
11:a3|function hello() {
22:f1|  return "world";
33:0e|}
```

Edits reference anchors instead of reproducing exact old text:

- replace line `22:f1`
- replace range `11:a3` through `33:0e`
- insert after `33:0e`

Benefits:

1. **Less brittle than exact `str_replace`.** The model does not need to reproduce whitespace, indentation, or unique old strings.
2. **Safer under concurrent changes.** If the file changes after reading, the hash mismatch rejects the edit before corruption.
3. **Less retry churn.** Fewer failed edits means fewer error messages, fewer re-reads, and fewer output tokens.

## Comparison with existing edit approaches

### Codex-style `apply_patch`

Diff-like freeform text works well for models trained on that format but fails badly elsewhere. Reported patch failure rates:

- **Grok 4:** 50.7%
- **GLM-4.7:** 46.2%

### Exact `str_replace`

Claude Code-like exact replacement requires exact old text and rejects multiple matches. Common failure: “String to replace not found in file.” This turns one intended edit into repeated context-expensive repair loops.

### Cursor-style merge model

Cursor uses a separate fine-tuned **70B** model to merge edits. The article notes Cursor’s claim that full-file rewrite can outperform aider-like diffs for files under **400 lines**. This is powerful but less portable than hashline because it requires separate model infrastructure.

## Evidence and caveats

The benchmark is controlled and task-specific, not peer-reviewed. It focuses on mechanical single-file bug restoration, not broad architecture tasks. The article summary does not include full result tables, confidence intervals, hash collision details, or behavior under formatter churn. Still, the token-saving result is credible in mechanism: failed edit loops are known to be expensive, and anchored edits reduce avoidable failures.

## Actionable takeaways for NeoKai

1. **Treat edit tools as performance infrastructure.** Measure edit failures, retry count, stale reads, non-unique matches, and token use per provider/model.
2. **Prototype hash-anchored edits.** NeoKai could expose read/grep line hashes and add line/range edit tools that reject stale anchors.
3. **Separate reasoning failures from tool-expression failures.** Telemetry should classify invalid patch syntax, stale anchor, non-unique replacement, and missing text distinctly.
4. **Optimize for multi-provider variance.** NeoKai supports many providers; edit syntax should not assume Claude-specific strengths.
5. **Save tokens by reducing retries.** The reported **61% output-token drop** for Grok 4 Fast suggests reliability work can be token-efficiency work.
6. **Normalize subagent outputs.** The article criticizes raw JSONL leakage; Space workflow messages should favor structured, concise artifacts over transcript dumps.

## Bottom line

The article shows high leverage at the tool boundary. For NeoKai, hash-anchored edits and harness-level benchmarking are strategic investments: they improve correctness and reduce context waste without changing models.
